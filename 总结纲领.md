# 总结纲领

## 目标
这份文件用于约束之后所有“总结 / 提炼 / 归纳 / 写成文档”的工作方式。

核心目标只有一个：

**把原始内容整理成可复盘、可执行、可快速理解的中文文档，而不是只做表面转述。**

以后每次你发给我的：

- 推文
- 长线程
- 评论区补充
- 聊天截图
- 图片里的文字
- 策略观点
- 研究框架
- 零散材料

默认都按这份纲领执行。

---

## 一、总原则

### 1. 优先提炼“作者真正想表达什么”
不要机械复述原文。

每次总结时，优先回答这些问题：

- 这篇内容的主题到底是什么？
- 作者真正的核心观点是什么？
- 作者的判断方法是什么？
- 作者提出了哪些应对策略？
- 作者回复 / 评论里的补充，是否比主帖更重要？
- 这套东西的适用场景、隐含前提和主要风险是什么？

### 2. 总结不是摘抄，而是重组
原文可能是散的、跳的、碎的、口语化的，甚至故意留白。

总结时必须做到：

- 去掉重复表达
- 合并相似观点
- 还原逻辑链条
- 把零散内容改写成结构化表达
- 把“作者怎么想”与“作者怎么做”分开写清楚

### 3. 默认写成投研笔记风格
默认风格更接近：

- 投研笔记
- 策略复盘
- 研究摘要
- 可执行卡片

而不是：

- 社交平台口语复述
- 流水账
- 情绪化转发文案

### 4. 结论必须比原文更清楚
总结后的文档必须做到：

**读总结的人，应该比直接看原文更容易明白作者到底在说什么。**

---

## 二、固定输出要求

以后只要是“写成文档”的任务，默认尽量包含下面这些部分。

### 1. 标题
标题优先使用**主题化标题**，而不是“某人某贴总结”。

好的标题应该直接说明内容主题，例如：

- `利用持仓人数比与资金费率识别 BTC 反转区间`
- `WTI-Brent价差套利复盘：判断逻辑与应对策略`
- `假流动性与动态成本：币安做市激励如何改变价格推动机制`

只有当原文主题特别不明确时，才允许用：

- `某帖子分析`
- `某聊天记录总结`

### 2. 原文信息
文档开头应尽量交代：

- 作者
- 原文链接（如果有）
- 时间（如果能拿到）
- 内容类型（主帖 / 长线程 / 评论补充 / 截图 / 聊天记录）
- 是否有配图

### 3. 主题
必须单独写一段，直接回答：

**这篇内容到底在讲什么。**

不能一上来就扎进细节。

### 4. 判断方法
如果原文涉及分析、交易、研究、观察框架，就必须单独整理：

- 作者怎么看
- 作者依据什么判断
- 作者用什么指标 / 现象 / 结构 / 前提来做判断

这是必须项。

### 5. 应对策略
如果原文涉及操作思路、执行方法、风险对冲、仓位处理、步骤安排，也必须单独整理：

- 作者建议怎么做
- 作者的操作顺序是什么
- 遇到极端情况如何处理

这也是必须项。

### 6. 风险与限制
如果内容带有策略性、交易性或强结论，原则上都要补一段：

- 这套方法依赖什么前提
- 失效点是什么
- 真正难点在哪里
- 纸面逻辑和实盘执行之间差什么

### 7. 扩散分析 / 延展思路
在完成原文总结之后，默认补一个“扩散分析 / 延展思路”环节。

这个环节的目标不是脱离原文乱发挥，而是：

- 基于原文的核心逻辑，往外延展出相邻思路
- 给出有启发性的变体、替代方案、升级版或降级版
- 帮助我从“理解作者”进一步走到“举一反三”

这个环节默认可以包含：

- 与原策略相近的替代策略
- 更激进的版本
- 更保守的版本
- 更容易执行但赔率更低的版本
- 更难执行但收益弹性更高的版本
- 原策略可能失效时的替代思路
- 能否迁移到其他标的、其他周期、其他市场

要求是：

- 必须以原文逻辑为起点
- 必须有启发性，不能只是换个说法重复原文
- 可以写“更好”的策略，也可以写“更差但更容易做”的策略
- 如果是交易类内容，要明确这些延展思路分别适合什么人、什么场景
- 如果只是猜想，要明确写出它是推演，不要伪装成作者原意

### 8. 一句话结论
结尾默认都要有一句压缩总结。

要求：

- 尽量一句话讲明白
- 能让人快速回忆这篇内容的核心

## 三、针对不同内容类型的特殊规则

### 1. 如果是推文 / 线程
必须先分清：

- 主帖是不是只有标题
- 真正正文是不是写在作者回复里
- 评论区里作者是否补了关键结论

如果作者自己的回复比主帖重要，必须明确写出来，例如：

- “主帖本身只有标题，真正的正文主要在作者后续回复里。”
- “评论区补充比主帖更重要，需要一起看。”

### 2. 如果是截图 / 聊天记录
不能只抄截图文字。

必须做：

- 先还原它在说什么策略或观点
- 再用完整语言重写
- 把口语化表达转成可读文档

如果截图里是碎片化聊天，应该主动整理成：

- 核心逻辑
- 判断方法
- 应对策略
- 案例
- 风险点

### 3. 如果原文带图
必须先判断图像属于哪种情况：

- 作者头像
- 正文配图
- 图表
- 聊天截图
- 数据截图

如果正文图很重要，默认要：

- 尽量把图加进文档
- 在文中标出图 1 / 图 2 / 图 3
- 必要时补一句“这张图在说明什么”

如果工具默认输出没把图带出来，但原文里实际有图，应该继续想办法把图找出来，而不是轻易下结论说“没有图”。

### 4. 如果作者在评论区补了很多内容
默认视为正文的一部分，不能忽略。

总结时必须额外回答：

- 评论区补充了什么新结论？
- 哪些关键点只在评论里出现？
- 如果只看主帖，会漏掉什么？

### 5. 如果内容涉及策略 / 交易 / 套利
默认都要特别写清：

- 赔率来源是什么
- 风险来源是什么
- 作者假设了什么前提
- 这是一套纸面逻辑，还是带执行门槛的东西

不能只写“很有道理”。

## 四、表达风格要求

### 1. 默认用中文写
原文是英文，也优先写成中文总结，除非你明确要求保留英文。

### 2. 清楚、直接、简洁
写法要满足：

- 清楚
- 直接
- 可快速理解
- 尽量少废话

不要为了显得专业而故意写得晦涩。

### 3. 可以结构化，但不要僵硬
允许使用标题、小节、要点。

理想状态是：

- 有结构
- 但读起来仍然像人在认真解释

### 4. 少用空话，多用解释
少写这种句子：

- “这个观点很有启发”
- “值得深思”
- “很有价值”

多写这种句子：

- “作者真正依赖的前提是……”
- “这套策略真正的风险不在方向判断，而在执行……”
- “如果只看主帖，会漏掉评论区里这一段关键补充……”

### 5. 能举例时尽量举例
当一个逻辑比较抽象时，尽量给一个简单例子，帮助理解。

特别是：

- 交易逻辑
- 对冲逻辑
- 风险来源
- 执行差异

## 五、默认结构模板

以后写总结文档时，优先参考下面这个顺序：

```markdown
# 标题

## 原文信息

## 主题

## 作者的判断方法

## 作者的应对策略

## 关键补充（如果作者在评论/回复中有补充）

## 风险与限制

## 扩散分析 / 延展思路

## 我的归纳 / 最直接的总结

## 一句话结论
```

如果内容更像复盘，也可以替换成：

```markdown
# 标题

## 原文附图 / 原文信息

## 主题

## 作者到底在做什么

## 作者的判断方法

## 作者的交易过程

## 作者的应对策略

## 风险点

## 扩散分析 / 延展思路

## 一句话结论
```

如果内容更像截图里的聊天策略，也可以替换成：

```markdown
# 标题

## 策略核心

## 原作者表达的判断逻辑

## 原作者给出的应对策略

## 案例

## 优点与缺点

## 扩散分析 / 延展思路

## 一句话结论
```

## 六、禁止事项

### 1. 不要只复述原文顺序
原文怎么写，不代表总结也要怎么写。

### 2. 不要忽略作者评论区补充
尤其是主帖很短、评论很长的时候。

### 3. 不要只写“观点”，不写“方法”
如果作者说了判断依据，就必须提炼出来。

### 4. 不要只写“方法”，不写“执行问题”
特别是交易类内容，必须写风险、门槛、兑现难点。

### 5. 不要轻易下结论说“没有图”
如果原文里看起来有图，要尽量继续确认。

### 6. 不要写成过度学术化空壳
除非原文本身就是正式论文，否则默认写成可读、可复盘的中文笔记。

---

## 七、回复呈现规则

以后在对话里给出总结结果时，也默认遵守下面这些要求：

- 解释要清楚、简洁
- 如果涉及代码，使用带正确语言标记的代码块
- 如果回答涉及多个主题，使用标题分段
- 合适时给出实用例子，帮助理解

这部分用于约束我在聊天中的呈现方式，不只约束落地文档。

## 八、以后默认执行规则

从这份文件开始，之后每次你让我：

- 总结一个帖子
- 分析一串评论
- 整理截图内容
- 提炼策略
- 写成文档

我都默认按这份纲领执行。

除此之外，新增一条固定规则：

- **每当完成一份资料的分析并落成文档后，默认提交一次 `git commit`。**
- **每当新增一份分析文档后，默认同步更新 `README.md`。**
- `README.md` 的更新规则是：
  - 按内容类型归类当前仓库文章
  - 为新增文档添加可点击链接
  - 用一句简短描述说明文档主要内容
  - 如果新增文档更适合放进已有分类，就补到对应标题下；如果现有分类不合适，再新增分类
- 这次提交的信息里，必须包含这份总结的大概内容，至少应尽量体现：
  - 主题 / 标的 / 资料核心方向
  - 作者名或来源关键词
  - 这份材料大致属于哪种总结（如：策略分析、线程整理、聊天截图总结、研究笔记等）
- 如果一次处理了多份资料，也应尽量在 commit 信息里概括本次整理的主要内容，而不是只写空泛的 `update`、`notes`、`fix`。
- 默认风格应尽量接近：
  - `总结 somBTC CBBTC/WBTC 跨链搬砖框架`
  - `整理 Wayne 延迟套利线程与评论分析`
  - `总结 ETH PUT 聊天截图策略与延展思路`

如果某次任务需要偏离这份纲领，你再单独说明。

## 九、当前版本的核心偏好

基于过往聊天，你当前更偏好下面这种输出：

- 标题直接点明主题
- 不只讲“讲了什么”，还要讲“怎么判断、怎么做”
- 作者评论区内容要重点关注
- 如果原文有图，尽量加图
- 文档要像投研笔记，不要像社交媒体复述
- 风险和执行难点不能省略
- 在总结之后，再补一个“扩散分析 / 延展思路”环节
- 延展分析要有启发性，能从原文推到相近策略、替代策略或更可执行的版本
- 每完成一份资料分析后，默认提交一次 `git commit`
- commit 信息里要带上这次总结的大概内容，不能写成空泛信息
- 最后要有一句话压缩总结
