# 总结纲领

## 目标

这份文件用于约束之后所有“总结 / 提炼 / 归纳 / 写成文档”的工作方式。

核心目标不变：

**把原始内容整理成更清楚、更可复盘、更方便后续使用的中文文档，而不是做表面转述。**

但从现在开始，这份纲领不再默认把所有材料都当成“投资策略”。

你后续发给我的内容，可能是：

- 投资 / 交易策略
- 套利机会与执行复盘
- 市场微结构分析
- 技术知识与工程实现
- 工具、面板、工作流
- 截图、聊天记录、评论串
- 多篇内容合并成一个专题

这些东西不能硬套同一套模板。

所以新规则是：

**先判断内容类型，再决定文档结构。**

---

## 一、先分类，再决定怎么写

开始整理之前，先判断这份材料主要属于哪一类。

### 1. 交易 / 投资策略类

重点是：

- 怎么判断
- 怎么做
- 风险在哪里

例如：

- 买 `Put`
- 卖波动率
- 资金费率 / OI / 因子框架

### 2. 套利 / 执行 / 基础设施类

重点是：

- 机会从哪里来
- 执行门槛是什么
- edge 到底来自哪里

例如：

- 延迟套利
- 跨市场搬砖
- 链上事件套利
- 低延迟交易基础设施

### 3. 技术知识 / 工程实现类

重点是：

- 这个技术问题到底是什么
- 作者如何实现
- 难点和边界在哪里

例如：

- 链上监听工具
- 数据面板
- 监控系统
- 自动化流程

### 4. 市场结构 / 观察框架类

重点是：

- 作者如何解释市场机制
- 哪些变量最关键
- 什么情况下框架失效

例如：

- 做市商行为
- 永续合约机制
- 庄币结构
- 流动性与订单流分析

### 5. 研究框架 / 方法论类

重点是：

- 作者如何研究
- 模型 / 框架解决什么问题
- 适合迁移到什么场景

例如：

- `PELT` 变点检测
- 因子研究框架
- 学习路径
- 研究方法

### 6. 工具 / 面板 / 工作流类

重点是：

- 这个工具到底解决什么问题
- 输入是什么
- 输出是什么
- 如何帮助决策

### 7. 多材料合并专题类

重点是：

- 多个来源分别提供了什么
- 它们之间的共识与分歧是什么
- 合在一起能形成什么更完整的框架

### 8. 截图 / 聊天记录 / 零散材料类

重点是：

- 先把碎片重写清楚
- 再判断它属于上面哪一类

如果一份材料同时跨了几类，先选一个**主类型**，再补必要模块，不要每种结构都来一遍。

---

## 二、总原则

### 1. 总结不是套模版，是重建逻辑

原文可能是散的、跳的、碎的、口语化的，甚至故意留白。

总结时要做的是：

- 去掉重复
- 合并相近观点
- 还原逻辑链条
- 把“作者看到什么”与“作者怎么做”拆开

### 2. 先回答“这到底是什么”

每次开始前，优先回答：

- 这篇东西到底在讲什么
- 它属于策略、套利、技术、工具，还是市场结构
- 作者真正想解决的问题是什么

不要一上来就进入细节。

### 3. 不要强行写不存在的部分

如果原文是技术实现，就不要硬写“作者的应对策略”。

如果原文只是概念解释，就不要硬写“交易过程”。

如果原文很短，也不要为了凑结构把一句话拉成一大段废话。

### 4. 评论区 / 回复 / 后续补充经常是正文

尤其是：

- 主帖只有标题
- 正文在回复里
- 评论区有更关键的修正
- 链接推文实际正文在 `article` 里

这些都必须主动识别，不能只抓表层。

### 5. 结论必须比原文更清楚

目标始终是：

**读总结的人，应该比直接看原文更容易明白作者到底在说什么。**

---

## 三、所有文档的固定必备项

不管内容类型是什么，新增分析文档时，默认至少应尽量包含下面这些部分。

### 1. 标题

优先使用**主题化标题**，而不是“某人某贴总结”。

标题应尽量直接说明主题，例如：

- `Arbitrum无本地节点套利：Aave ETH折价、FCFS排序与低延迟执行`
- `链上LP流动性监控器：用Uniswap V3_V4加减仓识别热点与策略者动向`
- `做市商库存报价系统：报价倾斜、点差扩张与散户逆价体验`

### 2. 原文信息

文档开头应尽量交代：

- 作者
- 原文链接
- 时间
- 内容类型
- 是否有配图 / 附图 / 外链文章

### 3. 原文归档

每次新增分析文档时，默认同时归档原始材料。

目录约定：

```text
sources/<slug>/
  original.md
  original.txt
  meta.json
  assets/
```

`<slug>` 应尽量体现：

- 作者或来源关键词
- 主帖 ID 或时间
- 主题关键词

### 4. 主题

必须单独写一段，直接回答：

**这篇内容到底在讲什么。**

### 5. 风险与限制

只要内容带有结论、方法、机制、执行门槛或可迁移性，原则上都要写：

- 前提是什么
- 哪些地方容易误读
- 哪些地方难执行
- 哪些边界一变就可能失效

### 6. 一句话结论

结尾默认用一句话压缩全文核心。

## 四、可选模块库

下面这些模块不是每篇都要全部出现，而是按内容类型选择。

### 1. 判断方法

适合：

- 策略类
- 市场观察类
- 研究框架类

重点写：

- 作者怎么看
- 用什么变量 / 指标 / 结构来判断

### 2. 应对策略

适合：

- 交易策略类
- 套利执行类
- 风险管理类

重点写：

- 怎么做
- 顺序是什么
- 什么时候不该做

### 3. 套利路径 / 机会来源

适合：

- 搬砖
- 延迟套利
- 链上事件套利
- 资金费率 / 价差类机会

重点写：

- 错价从哪里来
- 谁在出错
- 利润由什么条件支撑

### 4. 执行门槛 / 基础设施

适合：

- 延迟竞争
- 链上执行
- 机器人 / 低延迟场景
- 工程型工具

重点写：

- 速度要求
- 资本要求
- 节点 / RPC / 地理位置 / 网络要求
- 技术栈难点

### 5. 实现思路 / 系统设计

适合：

- 技术文
- 工具文
- 工程实现

重点写：

- 问题定义
- 数据来源
- 处理流程
- 关键模块
- 已实现与待优化部分

### 6. 关键补充

适合：

- 评论区有重要修正
- 作者后续回复比主帖更关键
- 多个来源需要拼接

### 7. 扩散分析 / 延展思路

默认**强烈建议保留**，但不再要求每篇都写得很重。

它的目标是：

- 基于原文逻辑继续往外推
- 给出更激进 / 更保守 / 更容易执行 / 更难但更强的变体
- 帮助从“理解作者”走到“举一反三”

如果材料特别短、信息密度不够，延展可以写得短一点，但不要为了形式硬凑。

### 8. 适用场景 / 边界条件

适合：

- 策略类
- 套利类
- 技术工具类
- 研究框架类

重点写：

- 适合什么市场
- 适合什么人
- 什么情况下不适合

## 六、针对不同来源的特殊规则

### 1. 如果是推文 / 线程

必须先分清：

- 主帖是不是只有标题
- 正文是不是在作者回复里
- 链接推文的正文是不是在 `article`
- 评论区里作者有没有补关键结论

如果作者自己的回复比主帖重要，必须明确写出来。

### 2. 如果是截图 / 聊天记录

不能只抄截图文字。

必须做：

- 先还原它在说什么
- 再用完整语言重写
- 把口语化表达转成可读文档

### 3. 如果原文带图

必须先判断图像属于哪种情况：

- 正文配图
- 图表
- 聊天截图
- 数据截图
- 单纯头像或装饰图

如果图很重要，默认要：

- 归档到 `assets/`
- 尽量在 `original.md` 中保持相对位置
- 在分析文档里解释这张图在说明什么

### 4. 如果作者在评论区补了很多内容

默认视为正文的一部分。

总结时必须额外回答：

- 评论区补了什么
- 哪些关键点只在评论里出现
- 如果只看主帖会漏掉什么

### 5. 如果内容涉及策略 / 交易 / 套利

默认要特别写清：

- edge / 赔率来源是什么
- 风险来源是什么
- 执行门槛是什么
- 这更像纸面逻辑，还是更像基础设施竞争

### 6. 如果内容更偏技术 / 知识 / 工程

不要硬写：

- 收益率
- 仓位处理
- 对冲建议

应该改写成：

- 问题是什么
- 为什么这样设计
- 实现难点是什么
- 误区和边界是什么

## 七、表达风格要求

### 1. 默认用中文写

原文是英文，也优先写成中文总结，除非你明确要求保留英文。

### 2. 清楚、直接、简洁

写法要满足：

- 清楚
- 直接
- 可快速理解
- 尽量少废话

### 3. 可以结构化，但不要僵硬

理想状态是：

- 有结构
- 但结构服务内容
- 不要让每篇文档都像同一台机器印出来的

### 4. 少用空话，多用解释

少写：

- “很有启发”
- “值得深思”
- “很有价值”

多写：

- “作者真正依赖的前提是……”
- “这套东西最难的不是方向，而是执行……”
- “这里真正有用的不是指标本身，而是……”

### 5. 能举例时尽量举例

特别是：

- 套利路径
- 低延迟执行
- 工具面板用途
- 技术实现边界

### 6. 明确区分“作者原意”和“我的延展”

如果是推演、升级版、替代版或个人补充，必须让读者看得出来，不要伪装成作者原话。

## 八、README、归档与 git 规则

### 1. 新增分析文档时，默认同步归档原文

归档内容尽量包括：

- 原文正文
- 原文图片 / 图表 / 截图
- 作者、链接、时间等元信息

### 2. 新增分析文档时，默认同步更新 `README.md`

`README` 更新要求：

- 放到最合适的已有分类下
- 如果已有分类不合适，再新增分类
- 添加可点击链接
- 用一句短描述说明主题、来源和核心内容

### 3. 不是每次微小修改都必须改 README

下面这些情况通常**需要**更新 `README.md`：

- 新增分析文档
- 新增专题
- 仓库分类发生变化

下面这些情况通常**不需要**单独改 `README.md`：

- 修正文档错字
- 修补链接
- 更新 `meta.json`
- 只修改 `总结纲领.md`

除非 `README` 的仓库说明本身也需要同步调整。

### 4. 每次完成一批相关文件修改后，默认提交一次 `git commit`

这里的“相关文件修改”包括但不限于：

- 新增分析文档
- 归档原文和配图
- 更新 `README.md`
- 修改 `总结纲领.md`
- 修正文档内容、标题、链接或格式

### 5. 提交前默认先检查当前变更

提交前要确认：

- 只提交本次相关文件
- 不顺手带上无关修改

### 6. 提交信息必须参考仓库历史风格

要求：

- 用清楚的中文短句
- 不能写成空泛的 `update`、`fix`、`notes`
- 尽量体现主题 / 来源 / 修改方向

风格应尽量接近：

- `总结 hklst4r Arbitrum无本地节点套利与低延迟执行`
- `总结 MonkeyJcck 链上LP监控与热点识别工具构想`
- `总结 多篇山寨合约盘结构与OI信号失真`
- `更新总结纲领：按内容类型拆分模板与写法`

## 九、禁止事项

### 1. 不要把所有内容都写成交易策略

技术文、工具文、知识文，不能硬套“判断方法 + 应对策略 + 仓位处理”。

### 2. 不要为了统一而强行统一结构

结构应该服务内容，不是内容服务结构。

### 3. 不要只复述原文顺序

原文怎么写，不代表总结也要怎么写。

### 4. 不要忽略评论区和后续修正

尤其是主帖很短、正文很长的时候。

### 5. 不要只写“观点”，不写“机制”

如果作者讲了为什么、怎么成立、靠什么成立，就必须提炼出来。

### 6. 不要只写“方法”，不写“门槛和边界”

特别是套利和技术类内容，必须写执行难点和限制。

### 7. 不要轻易下结论说“没有图”或“没有正文”

如果原文看起来有外链文章、配图、回复正文，要继续确认。

### 8. 不要把自己的延展写成作者原话

这是最容易让文档失真的地方之一。

---

## 十、当前默认偏好

基于过往聊天，当前默认偏好是：

- 标题直接点明主题
- 先判断内容类型，再选合适结构
- 优先解释“到底在讲什么”和“为什么成立”
- 如果是策略 / 套利，重点写判断方法、执行门槛、风险来源
- 如果是技术 / 工具，重点写问题定义、实现思路、边界条件
- 作者评论区和后续补充要重点关注
- 如果原文有图，尽量保留并解释
- 文档要像研究笔记，不要像社交媒体复述
- 风险、限制、执行难点不能省略
- 默认尽量补一个有价值的延展思路
- 新增分析文档后要更新 `README.md`
- 完成一批相关文件修改后默认提交一次 `git commit`
- 最后保留一句话结论

如果某次任务需要偏离这份纲领，再单独说明。
