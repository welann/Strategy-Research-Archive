## 主题

这篇文章复盘了作者在 Polymarket 的一次策略迁移：

- 原先做吃单套利，因为二元市场两条腿不能保证原子成交，裸腿亏损约 `800 USDC`；
- 后来利用 maker 费用和返佣优势，转向 BTC 15 分钟涨跌市场的双边挂单；
- 第一版机器人在平稳行情中持续赚取价差；
- 趋势行情到来后，订单流只成交即将亏损的一侧，库存迅速偏斜；
- 系统表面的 `NoArb` 指标仍然很好看，但未配平库存已经成为裸方向仓。

文章最重要的结论不是“maker 比 taker 更赚钱”，而是：

**二元市场做市的收益来自配平库存，风险来自配不平的剩余库存；报价决定毛利，逆向选择和库存控制决定策略能否活下来。**

作者将大部分工程工作用于库存修复、超卖保护和停止条件，而不是挂单逻辑。这也是做市系统与简单价差脚本的根本区别。

## 机会来源

### 1. 吃单套利和做市承担的是不同风险

吃单套利的目标是同时买齐一组完整结果：

```text
Up 成本 + Down 成本 < 1
```

如果两条腿都成交，结算时必有一边兑付 `1`，利润被锁定。

问题在于，多腿订单如果没有真正原子成交：

- 一条腿成交；
- 另一条腿失败或价格移动；
- 原本的套利立即变成方向仓。

短周期市场留给补腿的时间非常少，因此作者曾将一小时以下市场拉黑。

双边做市则不是主动抢现成价差，而是在两边挂 maker 买单，等待其他交易者卖给自己。它的优势来自：

- 不支付 taker 手续费；
- 可能获得 maker 返佣；
- 被动成交能捕获买卖价差；
- 15 分钟市场换手快、轮次多。

但 maker 不是无风险地领取价差。价差本质上是市场支付给做市方承担库存风险和信息劣势的补偿。

### 2. 二元合约提供天然的配平结构

BTC 15 分钟涨跌市场有两个互斥且完备的结果：

- `Up`
- `Down`

结算时：

```text
Up + Down = 1
```

如果持有等量的两边份额，无论结果如何，每一对都会兑付 `1`。

这为做市方提供了清晰的配平目标：分别吸纳两边库存，并把一对的总成本压到 `1` 以下。

### 3. Maker的利润来自急单和情绪单跨过价差

愿意立即成交的订单可能来自：

- 追涨杀跌；
- 临近结算时的概率重估；
- 对冲需求；
- 信息反应；
- 仓位退出。

这些交易者愿意牺牲价格换取速度，做市方则通过被动报价获取价差。

但同一批订单也可能携带信息优势。交易越急、越单边，maker 越需要怀疑自己正站在错误的一侧。

## NoArb数学

### 1. 等量库存时，WAC之和可以描述锁定利润

定义：

- `q_U`：Up 数量；
- `q_D`：Down 数量；
- `WAC_U`：Up 加权平均成本；
- `WAC_D`：Down 加权平均成本。

当两边数量相等：

```text
q_U = q_D = q
```

忽略手续费和返佣，结算利润为：

```text
PnL = q × [1 - (WAC_U + WAC_D)]
```

作者把：

```text
NoArb = WAC_U + WAC_D
```

作为核心指标。

因此：

- `NoArb < 1`：等量配平部分有正结算利润；
- `NoArb = 1`：配平部分不赚不亏；
- `NoArb > 1`：配平部分锁定亏损。

原文例子：

```text
WAC_U = 0.47
WAC_D = 0.44
NoArb = 0.91
```

每一对份额的毛利润为：

```text
1 - 0.91 = 0.09
```

### 2. 数量不等时，NoArb不能代表整个账户

真正重要的配平数量是：

```text
m = min(q_U, q_D)
```

只有 `m` 对库存能保证结算兑付。

整个库存的总成本是：

```text
C = q_U × WAC_U + q_D × WAC_D
```

在不知道最终结果时，库存组合的最低兑付为：

```text
Guaranteed payout = min(q_U, q_D)
```

因此忽略返佣后的保证盈亏下界是：

```text
Guaranteed PnL = min(q_U, q_D) - C
```

当两边等量时，它退化为作者的 `NoArb` 公式。

当两边不等量时，较多一侧的剩余库存没有保证兑付。此时即便：

```text
WAC_U + WAC_D < 1
```

账户整体仍可能亏损。

### 3. 更应该同时看三个指标

单独看 `NoArb` 不够，至少需要同时监控：

**配平利润：**

```text
matched_edge = 1 - WAC_U - WAC_D
```

**库存偏斜：**

```text
imbalance = q_U - q_D
```

**总成本相对最低兑付的缺口：**

```text
guaranteed_pnl = min(q_U, q_D) - total_cost + rebates - fees
```

进一步还应按当前公平概率对未配平库存做盯市，而不能只等结算。

### 4. 返佣不能修复错误的库存结构

Maker 返佣可以提高每笔成交的边际收益，但它通常只能抵消：

- 小额价差损耗；
- 部分不利成交；
- 交易基础设施成本。

如果未配平库存最终归零，方向亏损可能远大于返佣。

因此返佣是策略收益的一部分，不是风险控制。

## 第一版为何失效

### 1. 对称报价不代表对称成交

第一版机器人的假设是：

- Up 和 Down 两边对称挂单；
- 看涨和看跌交易者都会出现；
- 两边库存会自然接近平衡。

这个假设在平稳行情里看起来成立，因为订单流相对双向。

但在 BTC 突然上涨时：

- 交易者急于卖出 Down；
- 作者的 Down 买单连续成交；
- 没有人愿意卖出 Up；
- Down 库存快速累积；
- 配平所需的 Up 库存没有出现。

对称挂单只描述做市方的意图，不能保证市场给出对称成交。

### 2. 逆向选择让maker专门收到“坏库存”

逆向选择的核心是：

**愿意主动跨过价差与你成交的人，平均可能比你更清楚当前价格已经过时。**

趋势行情中，maker 容易发生：

- 在价格即将下跌前买到较贵的资产；
- 在概率快速重估时持续承接失败结果；
- 撤单速度慢于信息交易者的吃单速度；
- 成交越多，库存方向越错误。

因此，成交量上升不一定是好事。

如果成交突然单边涌入，它可能不是“生意变好”，而是报价已经变成市场上最便宜的错误。

### 3. 低成本库存可能制造账面幻觉

趋势上涨时，Down 价格不断下跌，机器人可能持续以更低价格买入 Down。

这会让：

- `WAC_D` 下降；
- 表面的 `NoArb` 变好；
- 单笔成交看起来越来越便宜。

但如果没有相应 Up 库存，低价 Down 仍可能在结算时全部归零。

“越跌越便宜”并不能把单边库存变成套利仓。

### 4. 做市裸腿比吃单裸腿更隐蔽

吃单套利裸腿通常很明显：

- 一腿成交；
- 一腿失败；
- 系统立即暴露方向风险。

做市裸腿则逐笔形成：

- 每笔订单都按预设价格正常成交；
- 没有单笔明显错误；
- 库存偏斜不断累积；
- 直到行情不回头，方向风险才完整暴露。

因此做市系统需要持续的组合级风险判断，不能只检查订单级成功。

## 库存管理框架

### 1. 报价应该随库存倾斜

当 Down 库存过多时，系统不应继续对称报价。

它可以：

- 降低或撤掉 Down 买单；
- 提高对 Up 的买入积极度；
- 挂出 Down 卖单释放库存；
- 缩小目标仓位；
- 提高继续承接 Down 所需的风险溢价。

这类 `quote skew` 的目标不是预测方向，而是让新增成交更有利于恢复库存平衡。

### 2. 设定硬性库存上限

至少应限制：

- 单边最大份额；
- 单边最大成本；
- 偏斜绝对值；
- 偏斜占总库存比例；
- 单轮最大亏损；
- 多轮累计风险。

超过阈值后，系统应降速、只减仓或停止，而不是继续等待“另一边迟早会成交”。

### 3. 把毒性订单流作为独立信号

可监控：

- 短时间成交方向比例；
- 标的 BTC 的瞬时收益和波动率；
- 报价被吃后的价格漂移；
- 成交后若干秒的 markout；
- 撤单到确认的延迟；
- 一侧成交速度相对另一侧的倍数。

如果 maker 在成交后持续遭遇负 markout，说明报价正被更快或信息更强的交易者选择。

### 4. 越接近结算，库存风险越难修复

15 分钟市场的剩余时间不断缩短。

接近结算时：

- 概率对标的价格更敏感；
- 错误一侧更快趋近零；
- 可用于等另一侧成交的时间更少；
- 主动平仓滑点可能更大；
- 新增库存的修复窗口更短。

所以仓位上限和报价宽度应随剩余期限动态变化，而不是整轮固定。

### 5. 修不动就停

作者把系统原则概括为：

1. 不让库存歪；
2. 歪了尽快修；
3. 修不动就停。

第三条最重要。

如果主动卖出会锁定过大亏损、对侧没有深度、行情仍在加速，继续报价通常只会扩大问题。

此时应进入明确的故障状态：

- 取消所有新增订单；
- 冻结新的风险；
- 评估主动退出和持有结算的损失；
- 记录触发原因；
- 防止自动恢复后立刻重新积累同类库存。

## 执行与基础设施

### 1. 订单状态必须以交易所确认为准

系统需要处理：

- 部分成交；
- 撤单与成交竞态；
- 重复事件；
- WebSocket 丢包和重连；
- 本地状态与交易所状态不一致；
- 下单响应成功但订单实际被拒；
- 订单已成交但本地尚未收到更新。

库存控制依赖准确状态，一次漏记成交就可能让后续报价方向完全错误。

### 2. 回测很难重现真实排队和逆向选择

仅使用 K 线或盘口快照，通常无法准确模拟：

- 队列位置；
- 哪些订单会先成交；
- 撤单延迟；
- 行情跳跃时的穿透成交；
- 同一毫秒内成交和撤单的先后；
- maker 返佣的实际计算；
- 信息交易者选择哪一侧报价。

作者也提到，有 bug 只在剧烈实盘行情中出现，在模拟环境里几乎不可复现。

因此策略需要：

- 小资金实盘；
- 完整事件日志；
- 逐笔回放；
- 故障注入；
- 最坏路径测试；
- 与真实成交结果对账。

### 3. 监控面板不能只显示利润

至少应实时显示：

- 两边数量和成本；
- 配平数量；
- 未配平库存；
- `NoArb`；
- 保证盈亏下界；
- 按当前概率估值的组合 PnL；
- 单边成交速率；
- 成交后 markout；
- 剩余结算时间；
- 当前机器人状态。

如果面板只显示 `NoArb` 和累计返佣，很容易掩盖方向风险。

## 风险与限制

### 1. 原文没有披露足够长的实盘收益

作者明确表示，要等小资金实盘样本足够长后再披露：

- 平均 `NoArb`；
- 每轮收益；
- 长期表现。

因此目前能确认的是机制和失败案例，不能确认策略已经具有稳定正期望。

### 2. 费率和返佣政策会变化

文章提到 50 美分附近 taker 手续费峰值约 `1.8%`，maker 零费并有返佣。

这属于作者写作时的市场条件。平台可能调整：

- 费率曲线；
- 返佣来源；
- 适用市场；
- 资格规则；
- 活动周期。

任何实盘策略都应从官方接口读取当前费率，而不是把文章数字硬编码。

### 3. 高换手同时意味着高毒性

短周期市场提供更多轮次，但也意味着：

- 信息变化快；
- 撤单窗口短；
- 标的跳动更容易让报价过时；
- 临近结算时 gamma 式敏感度更高；
- 单轮错误来不及通过未来轮次补救。

“流水线”只有在库存控制有效时成立，否则只是更高频率地重复风险。

### 4. 学术套利规模不等于个人可获得容量

原文引用的 arXiv 论文估算其研究样本中约有 `4000 万美元`已实现套利利润。

这说明定价无效率真实存在，但不能推出：

- 当前仍有同等规模；
- 这些利润来自同一种 15 分钟双边做市；
- 普通机器人能够获得相同订单流；
- maker 是利润获得者而不是被套利者。

论文研究的是更广泛的市场重平衡和组合套利，个人策略仍需独立验证。

### 5. 二元市场仍有规则和结算风险

除了交易执行，还要关注：

- 市场问题定义；
- 数据源；
- 截止时间；
- 平局或异常价格处理；
- 结算争议；
- 平台和链上基础设施风险。

对于 15 分钟价格市场，任何时间戳或数据源理解错误都可能直接改变结果。

## 扩散分析 / 延展思路

### 1. 用保证盈亏下界替代单独的NoArb

`NoArb` 适合描述等量配平部分，但风险引擎更应该优先计算：

```text
min(q_U, q_D) - total_cost + rebates - fees
```

这个指标直接回答：

**如果现在停止交易，并按最差结果结算，账户至少会赚或亏多少。**

### 2. 将库存风险按剩余时间缩放

同样数量的不平衡库存：

- 距离结算 `14` 分钟时，仍可能通过成交修复；
- 距离结算 `30` 秒时，几乎就是方向下注。

风险限额应同时依赖：

- 库存偏斜；
- 标的波动率；
- 剩余时间；
- 当前概率；
- 对侧盘口深度。

### 3. 建立maker毒性评分

可以根据每笔成交后的 markout 估计订单流毒性：

```text
toxicity = 成交后公平价值变动对maker的不利程度
```

如果某一侧持续出现负 markout：

- 扩大该侧价差；
- 降低挂单量；
- 缩短订单存活时间；
- 暂停该侧报价；
- 提高恢复报价的确认门槛。

### 4. 把机器人设计成状态机

与其让多个条件零散控制订单，更适合明确状态：

- `NORMAL`：双边报价；
- `SKEWED`：库存偏斜，报价倾斜；
- `REDUCE_ONLY`：只允许减仓；
- `TOXIC_FLOW`：检测到毒性订单流，暂停；
- `CLOSE_ONLY`：临近结算，只处理现有库存；
- `HALTED`：数据或交易状态异常，全部撤单。

状态机能减少复杂条件互相冲突，也便于事故复盘。

### 5. 将策略收益拆成不同来源

不要只看总 PnL，应拆分：

- 买卖价差；
- maker 返佣；
- 配平套利收益；
- 未配平库存方向收益；
- 主动修仓损益；
- 手续费和滑点；
- 异常执行损失。

只有拆开后，才能判断系统真正赚的是做市 edge，还是偶然押对方向。

## 一句话结论

**Polymarket 双边做市只有等量配平部分才是 `NoArb`，未配平库存仍是方向仓；真正的核心不是把两边报价挂出去，而是识别毒性订单流、约束库存偏斜，并在无法修复时及时停止。**
